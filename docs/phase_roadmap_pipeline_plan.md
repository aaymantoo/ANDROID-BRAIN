## 1. The 2026 Enterprise Context

### How Leading Teams Build Android Apps with AI in 2026

In 2026, enterprise Android teams no longer write boilerplate — they orchestrate it. The dominant workflow is:

```
PM writes PRD
    │
    ▼ AI enrichment tool (Brain Engine Phase 0B)
Hyperspec PRD with full behavioral specs
    │
    ▼ Brain init
PROJECT_BRAIN.json  +  ROADMAP.md
    │
    ▼ Agent session (Claude Code or Antigravity CLI + MCP)
Feature-by-feature Kotlin generation, validated, committed
    │
    ▼ Next session picks up exactly where the last left off
Complete production app
```

**Three tools dominate this workflow:**

| Tool | Role | MCP Config |
|---|---|---|
| **Claude Code** (`claude`) | Primary orchestrator, recommended | `.claude/settings.json` |
| **Antigravity CLI** (`antigravity`) | Google's agent CLI, replaced Gemini CLI at I/O 2026. Faster (Go), same harness as Antigravity IDE | `~/.gemini/config/mcp_config.json` |
| **Antigravity IDE** | VS Code-based desktop agent, shares config with CLI | Same `mcp_config.json` |

At Google I/O 2026, Google announced Antigravity 2.0 — a standalone agent-first platform with CLI, SDK, managed execution, and enterprise support. Antigravity CLI and Claude Code both support MCP servers identically: they call tools via stdio, pass arguments as JSON, and receive structured responses.

**The missing piece** in every existing solution: **session continuity**. When a developer closes their terminal and opens a new one the next day, the agent has zero memory of what was built. They must manually re-explain context. Brain Engine Phase 0C eliminates this entirely with a persistent, self-updating `ROADMAP.md` and dedicated "resume session" MCP tools.

## 2. What This Phase Adds

### The Core Problem

After Phase 4, Brain Engine can generate any component for any screen. But:

- It has no concept of **features** (auth, home, orders, profile are just flat screens)
- It has no concept of **"what was built last session"**
- It has no concept of **"what should be built next"**
- `generation_history` exists but is a raw append-only log — unusable as a session resume point
- A new session starts blind

### The Solution: Three Interlocking Pieces

```
PROJECT_BRAIN.json          ROADMAP.md                  MCP Tools
─────────────────           ──────────────              ─────────────────────
Feature groups         →    Human-readable         →    get_session_context()
GenerationStatus            progress board              get_next_task()
per screen/component        readable without            get_feature_status()
                            MCP server running          get_project_roadmap()
                                ▲
                                │ auto-updated after every generation
```

## 5. Schema Changes (`project_brain/brain/schema.py`)

### New models

```python
class Feature(StrictModel):
    id: str                              # "auth", "home", "orders", "profile"
    name: str                            # "Authentication", "Home Feed"
    description: str = ""
    screens: list[str]                   # screen_ids belonging to this feature
    priority: int = 0                    # build order (1 = first)
    status: str = "planned"              # planned | in_progress | complete
    feature_dependencies: list[str] = Field(default_factory=list)  # feature_ids

class ComponentStatus(StrictModel):
    viewmodel: bool = False
    ui_state: bool = False
    repository: bool = False
    scaffold: bool = False
    di_module: bool = False
    nav_route: bool = False
    tests: bool = False
    validated: bool = False

    @property
    def all_complete(self) -> bool:
        return all([
            self.viewmodel, self.ui_state, self.repository,
            self.scaffold, self.di_module, self.nav_route,
            self.tests, self.validated,
        ])

class GenerationStatus(StrictModel):
    screen_id: str
    components: ComponentStatus = Field(default_factory=ComponentStatus)
    last_generated: str | None = None    # ISO timestamp
    last_validated: str | None = None
    violation_count: int = 0

class SessionEntry(StrictModel):
    date: str                            # ISO date
    summary: str
    components_built: list[str] = Field(default_factory=list)
    feature_completed: str | None = None
```

### Additions to `ProjectBrain`

```python
class ProjectBrain(StrictModel):
    # ... existing fields ...
    features: list[Feature] = Field(default_factory=list)
    generation_status: list[GenerationStatus] = Field(default_factory=list)
    session_log: list[SessionEntry] = Field(default_factory=list)
    roadmap_updated_at: str | None = None
```

## 6. New Files

### `project_brain/generators/roadmap_generator.py`

```python
class RoadmapGenerator:
    def generate(self, brain: ProjectBrain) -> str:
        """Produce the full ROADMAP.md content."""

    def write(self, brain: ProjectBrain, path: Path) -> None:
        """Write ROADMAP.md alongside PROJECT_BRAIN.json."""

    def update_after_generation(
        self,
        brain: ProjectBrain,
        screen_id: str,
        tool_name: str,
        success: bool,
    ) -> None:
        """Called by GenerationOrchestrator after each generation. Updates
        GenerationStatus, checks if feature is now complete, appends to session log."""

    def _feature_progress(self, brain: ProjectBrain, feature: Feature) -> dict:
        """Return completion counts and next recommended step for a feature."""

    def _next_step(self, brain: ProjectBrain) -> str:
        """Infer the single most valuable next generation call."""
```

### `project_brain/tools/roadmap_tools.py`

Four new MCP tools:

```python
def get_session_context(brain: ProjectBrain) -> dict:
    """Return last session summary + recommended next step.
    Called at the start of every new session."""
    return {
        "last_session": brain.session_log[-1] if brain.session_log else None,
        "overall_progress": f"{done}/{total} components",
        "current_feature": current_feature_id,
        "next_step": next_recommended_call,
        "blocked_features": [...],
    }

def get_next_task(brain: ProjectBrain) -> dict:
    """Return single most important next generation step with exact tool call."""
    return {
        "tool": "generate_viewmodel",
        "args": {"screen_id": "home"},
        "reason": "Home ViewModel is the first unbuilt component in the current feature.",
        "feature": "home",
    }

def get_feature_status(brain: ProjectBrain, feature_id: str) -> dict:
    """Return component-level status for all screens in a feature."""

def get_project_roadmap(brain: ProjectBrain) -> dict:
    """Return full feature → screen → component status tree."""
```

### `project_brain/generators/prd_enricher.py` update

`PRDEnricher._extract_features()` — parse the enriched PRD to extract feature groupings.
The LLM already structures the hyperspec by feature section; this method maps screen IDs to feature IDs.

## 8. CLI Updates

### `brain init` — updated

```bash
brain init --from-prd enriched_prd.md
```

After generating `PROJECT_BRAIN.json`, automatically:
1. Extracts features from brain (via `_extract_features()`)
2. Initialises `GenerationStatus` for every screen
3. Writes `ROADMAP.md` alongside `PROJECT_BRAIN.json`

Output:
```
Brain written to: PROJECT_BRAIN.json
Roadmap written to: ROADMAP.md
Features: 4 (auth, home, orders, profile)
Screens:  8
Components to generate: 28
Start with: get_session_context()
```

### `brain roadmap` — new command

```bash
brain roadmap               # print current roadmap to terminal
brain roadmap --update      # regenerate ROADMAP.md from current brain state
brain roadmap --feature auth  # show only auth feature detail
```

## 10. Integration Guide: Claude Code & Antigravity CLI

### Claude Code

```json
// .claude/settings.json (project) or ~/.claude/settings.json (global)
{
  "mcpServers": {
    "brain-engine": {
      "command": "brain",
      "args": ["serve"],
      "env": {
        "BRAIN_PATH": "${workspaceFolder}/PROJECT_BRAIN.json"
      }
    }
  }
}
```

Or via CLI:
```bash
claude mcp add brain-engine -- brain serve
```

**Recommended session opener prompt** (paste into CLAUDE.md):
```
At the start of every session, call get_session_context() and report:
1. What was built last session
2. What feature is current
3. What the next generation step is
Then proceed without asking.
```

### Antigravity CLI (Google I/O 2026)

Antigravity CLI replaced Gemini CLI. It reads from `~/.gemini/config/mcp_config.json`:

```json
{
  "mcpServers": {
    "brain-engine": {
      "command": "brain",
      "args": ["serve"],
      "env": {
        "BRAIN_PATH": "/absolute/path/to/PROJECT_BRAIN.json"
      }
    }
  }
}
```

Note: Antigravity uses `serverUrl` (not `url`) for HTTP-based servers. For stdio (Brain Engine uses stdio), the `command` + `args` form above is correct for both Claude Code and Antigravity.

### Android Studio (Gemini in Android Studio)

Android Studio added native MCP support in 2025. Add Brain Engine under:
`Settings → Gemini → MCP Servers → Add`

```json
{
  "command": "brain",
  "args": ["serve"],
  "env": { "BRAIN_PATH": "${PROJECT_DIR}/PROJECT_BRAIN.json" }
}
```

## 13. What This Unlocks

### For solo developers
- Open laptop, run one command (`get_session_context`), know exactly where you left off.
- Never re-explain project context to the agent.

### For enterprise teams
- Multiple developers, different features, no coordination overhead — each reads `ROADMAP.md`.
- CI/CD can read `ROADMAP.md` to gate deployments (all components validated = ship).
- PM reads `ROADMAP.md` without touching code.
- New team member onboards in minutes: `brain status` + read `ROADMAP.md`.

### For the Brain Engine system itself
- Completes the "input → output → track → resume" loop.
- Phases 5 (bug forecasting) and 6 (sync) both depend on accurate per-screen generation status — this phase provides it.
