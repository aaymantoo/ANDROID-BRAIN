### 3. Apply the Enterprise Pattern Catalog
For every ViewModel, Repository, and Screen you define, apply ALL of the following 20 patterns:

**ViewModel Patterns:**
P01: UI State as data class — Use `data class XUiState(val isLoading: Boolean = false, ...)` not sealed class
     Loading/Success/Error. Only use sealed class when the screen has exactly 3 mutually exclusive states
     with no shared fields.

P02: One-shot events via Channel — Navigation, snackbar, dialog triggers are NEVER part of UiState.
     Define `private val _events = Channel<XEvent>(Channel.BUFFERED)` and `val events = _events.receiveAsFlow()`.
     Send events with `viewModelScope.launch { _events.send(XEvent.Navigate) }`.

P03: Atomic state mutation — Always use `_uiState.update { it.copy(field = value) }` not
     `_uiState.value = _uiState.value.copy(...)`. This is thread-safe.

P04: Mutex for concurrent guards — Any function the user can tap multiple times before the first completes
     (login, payment, form submit) must be wrapped: `private val _mutex = Mutex()` + `_mutex.withLock {}`.
     Apply when the PRD has a business rule about preventing duplicate calls.

P05: SavedStateHandle for nav args — Screen parameters (IDs, tokens passed via navigation) come from
     `savedStateHandle.get<String>("key")`, not constructor parameters. This survives process death.

P06: Typed error sealed class — Each feature has `sealed class FeatureError` with specific subtypes.
     Include a `companion object { fun from(t: Throwable): FeatureError }` factory. Never use
     `message: String` as the only error carrier.

**Repository Patterns:**
P07: Result<T> at all boundaries — Every repository method returns `Result<T>` (use `runCatching {}`).
     Use cases return `Result<T>` by delegating. ViewModels call `.onSuccess {}` / `.onFailure {}`.
     NEVER have exceptions crossing the repository-ViewModel boundary uncaught.

P08: Flow<T> for reactive streams — Live data (auth state, real-time Firestore) returns `Flow<T>`,
     not suspend functions. Use `callbackFlow { awaitClose { listener.remove() } }` for Firebase listeners.

P09: Coroutine-first Firebase — Always use `tasks.await()` from `kotlinx-coroutines-play-services`.
     Never use `.addOnSuccessListener {}` / `.addOnFailureListener {}` callbacks inside coroutines.

P10: DataStore over SharedPreferences — All lightweight persistence uses `DataStore<Preferences>`.
     Inject as `DataStore<Preferences>`. Keys defined as `private val KEY = stringPreferencesKey("key")`.

**Screen / Composable Patterns:**
P11: collectAsStateWithLifecycle — Always `val state by viewModel.uiState.collectAsStateWithLifecycle()`
     not `collectAsState()`. Requires `lifecycle-runtime-compose` dependency.

P12: LaunchedEffect for events — Collect one-shot events inside `LaunchedEffect(Unit) { viewModel.events.collect {} }`.
     Do NOT use `collectAsState` on Channel flows.

P13: Navigation via lambda parameters — Screen composables receive `onNavigateToX: () -> Unit` parameters.
     The NavController calls these lambdas at the nav graph level. ViewModel never sees NavController.

P14: @Stable on UiState — Annotate each data class UiState with `@Stable` or `@Immutable` for Compose
     compiler performance. Immutable data classes with only val fields are automatically treated as stable.

P15: Content composable separation — Every screen has `fun XScreen(viewModel, onNav)` (stateful) and
     `private fun XContent(uiState: XUiState, onEvent: (XIntent) -> Unit)` (stateless, testable).

**Architecture Patterns:**
P16: @Singleton repositories — All repositories are `@Singleton`. Single shared instance, single source of truth.

P17: Use cases as single-function classes — Each use case has exactly one `operator fun invoke()`.
     This makes it mockable and keeps ViewModel constructors declarative.

P18: @Keep on all Firestore/serialized models — Prevents ProGuard from renaming fields used in Firestore.

P19: Explicit Firestore field names — Use `@PropertyName("field_name")` or constructor parameter names
     that exactly match Firestore document field names. Never rely on Kotlin property names.

P20: Centralised error mapping — Each layer converts exceptions to its own error type. Repositories
     catch low-level exceptions and return `Result.failure(RepositoryException(...))`. ViewModels
     map repository errors to `FeatureError`. The error boundary is always explicit.
